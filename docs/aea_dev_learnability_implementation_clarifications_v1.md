# AEA development learnability implementation clarifications v1

**Recorded before model execution or audit-summary inspection:** 2026-07-14T23:14:33Z  
**Parent protocol:** `aea-dev-learnability-v1`

These clarifications resolve implementation details left implicit in the frozen preregistration. They do not change partitions, endpoints, mappings, support thresholds, budgets, inference, gates, or decision rules.

1. **Motor-free grounding training.** The existing repository learner always instantiates all three encoders. For the learnability analysis it uses the `null` arm and `motor_weight=0.0`, so the optimized objective is exactly video-text contrastive loss. The zero motor tensor remains present only because the shared training batch schema expects it. Grounding evaluation uses the existing motor-free dataset and does not load or call the motor encoder.
2. **Endpoint-specific training rows.** Each fine, coarse, or high-motion grounding model trains only on rows eligible for that endpoint under the preregistered six-window/three-event-group rule. Fold grouping remains the fixed endpoint-specific `StratifiedGroupKFold` assignment.
3. **Coarse grounding scores.** Training remains paired RGB-to-original-ASR-transcript contrastive learning. At evaluation, each coarse category score is the arithmetic mean of cosine similarities to the fixed fine-action prompts for its eligible member actions. Pairwise comparisons operate between these five predeclared category scores. No coarse token is introduced, no observed score selects a member verb, and fine prompts use the existing repository `action_prompt` template.
4. **Fine and high-motion prompts.** Fine and semantic-high-motion 2AFC use the fixed eligible fine action names with the existing repository `action_prompt` template. High-motion is a subset evaluation, not a motion-magnitude-derived label.
5. **Capacity subset.** The 64-window positive-control subset is selected from the preregistered eligible fine-action development rows. Repeatedly choose the currently least-sampled action (ties: total eligible support, then action name), then its candidate in the currently least-sampled event group (ties: SHA-256 of `aea-capacity-v1|example_id`, then example ID), until 64 rows are selected.
6. **Reserve file isolation.** A separate development-only JSONL is materialized from the already frozen partition manifest. The runner accepts only that JSONL. It verifies its ID set against the manifest and rejects confirmation IDs/groups. Source-hash validation remains attached to the earlier partitioning step; model code does not open confirmation RGB or IMU paths.
7. **Infeasible donor derangements.** As frozen, any train- or test-side whole-window bijection with no self/same-event-group donor that is mathematically impossible is marked invalid. It is not replaced with donor reuse, a changed fold, or a relaxed group rule. Synchronized-versus-chance estimates may still be reported descriptively, but the paired IMU gate and hard integrity gate fail for that endpoint.
